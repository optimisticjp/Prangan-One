# Project memory and next steps

This file exists so a future Claude Code session (or a human) can pick this project up without re-deriving everything from the source. Read this before making architectural changes. Updated after each major session; the most recent session is summarized first.

## Project identity

**What this is:** Prangan One, "The Society OS," a Gujarati-first, multi-tenant society management platform. It started as a single-society demo for Rajhans Tower (a real 24-flat society in Surat), which remains the first real society on the platform, created from the owner console like any other, not hardcoded as the only one. Domain: pranganone.com. Support: care@pranganone.com.

**The core stance that shapes every screen:** the committee is the customer and the hero, residents are upside. A society must get full value from the committee side alone, with zero residents logged in. Resident self-service is a second adoption wave, never a dependency for the society to run.

**Current phase:** Almost everything in `docs/PRANGAN_ONE_ROADMAP.md` that can be built without a live Supabase project now exists as real, tested code: the full role model, the subscription lifecycle with a working write-guard, the complete owner backend, real (if locally-scoped) magic-link auth code, CSV import, receipt cancellation, code splitting, an error boundary, and the public marketing site. The schema is fully written for the target state. What's left is fundamentally about a live database existing: connecting it, running the roadmap's own hard checkpoint (a real second society, verified isolation), and then the two things that were always going to need it (file storage, and the membership-claim flow that only makes sense once real auth users exist).

**Who it's for:**
- **Residents:** check their bill, pay, complain, read notices, vote, RSVP to events.
- **Committee (admin):** the treasurer/secretary/president running the society day to day. Almost never a software person, see "Designing for the committee's actual skill level" below.
- **Accountant:** a narrower role focused on the money side and audit exports.
- **Owner:** creates and manages every society on the platform from `/owner/*`. This role can be more technical than the others, since it's the platform owner's own tool, not a committee-facing one.

## Feature (this session): email + password login, alongside Google and magic link, not replacing either

Requested directly: a genuine third way to log in, for anyone who'd rather use a password than wait for an email or use a Google account.

**Deliberately designed so a password can never be the only way in.** Nobody signs up with a password cold - there'd be no way to verify a fresh password-only signup is actually who they claim to be, the same reasoning that's applied to every other identity decision in this app. Instead, someone who's already verified their identity the normal way at least once (magic link or Google) can opt into setting a password from their own profile or settings page, for faster login the next time. This is the only path that ever creates a password.

**Three places, one reusable component.** A `SetPasswordCard` shows up on the resident profile page, the admin/committee/accountant settings page, and the owner's dashboard - same component, same behavior, only rendered for a real session (the local demo has no real account to set a password on).

**Forgot password is real, not a stub.** A dedicated `/auth/reset-password` page, reached through Supabase's own password-recovery email link - deliberately not reusing `/auth/callback`, since a recovery click is a different intent than a normal login even though both start with an email link. Once a new password is set there, it hands off to the exact same session-resolution `/auth/callback` already does for every other login path.

**No new Supabase dashboard setup needed for this one.** Password sign-in lives under the same "Email" provider magic link already uses - confirmed this directly before writing anything, rather than assuming. If magic link already works (it does), password sign-in works the moment this code is live.

**Tests:** 185 passing, up from 178. 9 new, covering the three new auth functions directly (password sign-in, setting a password, and the reset-email request), each checked for the success path, a wrong-credentials/rejected-password path, and the not-configured path, matching how every other auth function in this file was already tested.

## Hotfix 2 (this session): activeSociety could genuinely be undefined, and now it structurally can't be

Found immediately after fixing the first hotfix, from the same live testing session, once Google login actually worked. A real owner's first genuine login correctly fetched zero real societies (since none had been created yet) - and the app had never accounted for that being possible.

**Root cause.** `activeSociety` was computed as `db.societies.find(...) ?? db.societies[0]` - if `db.societies` is empty, both sides of that fall through to `undefined`. The local demo never surfaced this, since the seed data always guarantees at least one society exists. But a real session with zero real societies is a completely normal, expected state (the owner's very first login, before creating anything), and `activeSociety` becoming undefined meant every place in the app that reads a field off `society` assuming it's always a real object - and there are several such places, not one - was one click away from crashing.

**Fix, deliberately structural rather than a single guard.** `activeSociety` can no longer be undefined at all - it falls back to a minimal, safe placeholder society object (reusing the same one already built for the "start completely empty" deployment mode) instead of `undefined`. This closes the entire class of bug at once, rather than patching whichever specific line happened to be the first to crash - a different page or component hitting the same empty-societies state would have just crashed somewhere else next.

**Tests:** 178 passing, up from 177. 1 new, reproducing the exact real condition (a saved database with zero societies) and confirming `society` is always a real, safe object, never undefined, no matter what.

## Hotfix (this session): a real crash on the live deployment, found and fixed

Not a planned session, a live bug report. The owner's leads page was crashing outright on a real, deployed session with `TypeError: o.unmatchedLoginAttempts is not iterable`.

**Root cause, found from the actual browser error, not guessed at.** `loadDB()` checks a saved browser snapshot against a version number before trusting it. That version number was never bumped across all the sessions that kept adding new fields to the `DB` shape (documents, polls, vehicles, contacts, `unmatchedLoginAttempts`, and others) - so a browser holding an older saved snapshot, still correctly tagged with the same version number, passed the check and came back missing whatever had been added since. The leads page was the one place that happened to iterate directly over that specific field, so it was the one that crashed; other pages reading the same stale snapshot just silently showed old data instead, which is a real, separate, already-known gap, not this bug.

**Confirmed the actual mechanism before fixing anything.** The activity log screenshot from the live site showed a real, one-off impersonation session with a one-character test reason "j" - a real click, not seed data - which is what confirmed the dashboard really was running on locally-cached data rather than a genuinely broken database connection.

**Fix:** `loadDB()` now defensively backfills every array field on `DB` against a fresh baseline if it's missing, null, or not actually an array, regardless of whether the version check passes. Fields that were already present and correct are left exactly as they were. This isn't specific to `unmatchedLoginAttempts` - it protects against the exact same class of bug for any field, including ones added in future sessions, without needing the version number to be remembered and bumped correctly every time the shape changes.

**Tests:** 177 passing, up from 174. 3 new, reproducing the exact real failure (a saved snapshot missing the field entirely), a related case (the field saved as `null` rather than simply absent), and a check that fields already present and valid are left untouched, not silently overwritten.

## Session 20 summary (this session): the membership gap, and five more tables go real

Started by actually checking something flagged as uncertain at the end of the last session, rather than assuming. That check found a real, blocking gap, fixed first, before moving through vendors, parking, contacts, and polls.

**The real find: a resident's join request was invisible to the committee meant to approve it.** `/join` already wrote a genuine pending membership row to the real database. Nothing on the committee's side ever fetched it, and approving or rejecting one only ever touched local demo data. A real resident could submit a request that would sit there, real, correctly pending, and completely invisible to the person supposed to review it. Verified the whole real flow end to end with actual Postgres: a resident submits a join request with an email not on file, it comes back pending, a real admin session can see it, approves it, and it becomes active, all through the same functions the app itself now calls.

**Vendors, parking, and contacts are real now** - the three simplest remaining tables, following the exact same fetch-once-and-cache, write-through-in-the-background shape everything since flats and bills has used. Contacts specifically only got a fetch, not a write, since there's no way to add one anywhere in the app currently, real or demo - nothing to wire on that side yet.

**Polls needed more care than the three before it, for a reason worth explaining.** A poll's votes used to live as a single embedded map, one vote per flat, that any code could just count directly. Once individual votes are actually private (an earlier session's fix), an ordinary resident's own fetch of that map only ever contains their own vote, not everyone's - counting it directly would show the wrong totals on their own screen. Built a real Postgres function, `poll_results`, that aggregates counts without ever exposing whose vote is whose, callable by any member but only returning anything when the poll's own visibility setting allows it, and always available to management regardless. Verified this exact behavior directly: a resident sees the true, correct totals but only their own individual vote row; hiding a poll's results correctly returns nothing to a resident while management still sees everything.

**Database changes this session:** `poll_results()`, a new aggregate-only function, plus verification that no new RLS gaps existed for the other three tables (there weren't any, all of it was already correctly in place from earlier sessions, this was purely an application-layer gap of never having built the fetch and write functions).

**What's left:** events is the one remaining module, and it's genuinely bigger than everything moved this session combined - contributions, volunteers, and expenses are three separate related tables under one event, not one. Documents now work the same real way complaint photos, payment proof, and society logos already do - an actual uploaded file with real, tested access control, not a filename sitting in a text field with a note saying the real thing is still ahead.

## Session 18 summary (this session): societies go real, and a genuinely tricky Postgres+RLS lesson

Started from the honest note at the end of the last session: society logos couldn't go real because societies themselves weren't real yet. Fixed the actual blocker first, then finished what it unblocked.

**The real find, and it's bigger than logos.** `activeSociety` was computed as `db.societies.find(s => s.id === session.societyId) ?? db.societies[0]` - since `db.societies` only ever held the local demo seed data, a real resident or admin's society id never matched anything in that array, and silently fell back to the first demo society. This meant every real login was showing the wrong society's name, logo, maintenance default, due day, receipt prefix, tenant access, and module settings the entire time - even though that same person's own flats, bills, and complaints were already correctly real. Fixed by giving societies the same real fetch-and-cache treatment as everything else: a resident or admin's own society is fetched on real login, the owner's full list is fetched separately (their console doesn't have one "active society" the way everyone else does, it's an aggregate view across all of them).

**A missing insert policy, found by trying to actually use the feature it should have supported.** `societies` had `select` and `update` policies but no `insert` policy at all - meaning literally no one, not even the real owner, could create a new society through the real database. The owner console's "add society" wizard would have failed outright the first time anyone actually tried it for real. Added the missing policy, verified directly: a real owner session can create a society, a fresh check with no membership at all cannot.

**Logo upload works now, for both a brand-new society during onboarding and an existing one from the owner console.** Same ordering rule as complaint photos and payment proof before it: the society has to exist in the real database first, then the logo uploads, then the URL gets attached back.

**The genuinely useful lesson from this session, worth remembering for any future bucket with this shape.** Building this, the logo upload kept failing in a way that made no sense - even a hardcoded `with check (true)` policy failed. Spent real time isolating it down to a single minimal reproduction before finding the actual cause: Postgres checks an `insert ... returning` clause's output against select-level RLS, the same way it would check a real `select` - and since `society-logos` is a public bucket, the assumption going in was that it wouldn't need its own select policy at all (public buckets serve reads directly, bypassing RLS entirely for actual end users). That assumption was right for real reads, but wrong for the upload client's own `returning`-style confirmation of what it just wrote. Added a `society_logos_select` policy specifically to fix this, confirmed the exact before/after with real inserts, and wrote the reasoning directly into both the schema comment and the README so this doesn't have to be rediscovered the same way next time a public-write bucket comes up.

**Database changes this session:** `societies_insert` (new, owner-only), `society_logos_select` (new, for the `returning` reason above). Both verified against a completely clean, from-scratch database and role, not just reasoned about.

**Tests:** 148 passing, up from 143. 5 new, covering fetching a single society, fetching every society an owner can see, and confirming `updateSocietyReal` never sends the owner-only protected fields even when asked to (that protection was already enforced at the trigger level from an earlier session; this test confirms the application-side function itself is also careful about it, not just relying on the database to catch a mistake).

**Still open:** documents (the fourth storage bucket, still needs its own more complex four-tier permission model) and the fuller onboarding wizard. Both unchanged from the last two sessions' notes.

## Decisions log

**Session 2 additions:**

- **Cloud hosting: Cloudflare Pages, not Vercel.** Verified this directly rather than assume: Vercel's free Hobby plan is restricted by its own terms to non-commercial, personal use, and this becomes a paid client engagement the moment it's set up for Rajhans Tower. Cloudflare Pages' free tier explicitly allows commercial use, with unlimited bandwidth and no credit card required. Same git-deploy workflow either way.
- **Auth: email magic link, not phone OTP.** The person chose this explicitly after being shown the cost tradeoff. Real phone OTP in India needs TRAI DLT sender registration (roughly ₹5,900 + GST one-time, 3 to 7 working days to clear) plus a per-message cost, none of which is actually free. Supabase's built-in email magic link needs no separate provider and costs nothing on the free tier. The tradeoff: this needs residents to have and check an email address, which won't be true for every member, especially older ones. The Members page now collects an optional email per flat (owner and tenant separately) for exactly this reason, and CLAUDE_CODE_NEXT_STEPS keeps a fallback path where the committee can act on a resident's behalf if that resident never gets a working login.
- **Supabase free tier's 7-day auto-pause is a real operational risk**, not a footnote. A free project with no traffic for seven straight days pauses itself until someone manually resumes it from the dashboard. For a live committee's app, a quiet week (a holiday, low adoption at first) would mean the app just looks broken. Fix: a scheduled free keep-alive ping (GitHub Actions cron or an UptimeRobot check) every few days. This needs to be set up as part of the go-live checklist, see supabase/README.md.
- **Not everything is configurable.** Explicitly rejected a fully generic "any field, any color, any layout" system. Societies control branding (curated theme + logo), setup (maintenance amount, due day, receipt numbering), and which modules are on, nothing else. - **Designed for the committee's actual skill level, not a typical SaaS admin's.** A Rajhans-style treasurer or secretary is comfortable with WhatsApp and a UPI app, not with SaaS jargon. Concretely: the onboarding wizard defaults everything to sensible values rather than starting blank; internal vocabulary (module, tenant, theme key) never appears in any user-facing copy, only in code comments and this file; the branding step is tap-to-choose preset cards, never a hex input; every module toggle has a one-line plain-language reason, not just a name. The onboarding wizard only collects a single support phone number on the Society record itself. Flagged here rather than silently skipped, this is a small, contained follow-up whenever it's needed.

**Session 1 decisions** (unchanged, still accurate): localStorage over a mock backend for the first pass so the committee could react to the UI before any backend investment; no gate/visitor security module, explicitly out of scope; spoken Surat Gujarati throughout, not textbook Gujarati; green/amber/red as the only status language app-wide; WhatsApp via `wa.me` share links rather than the Cloud API, since the Cloud API needs a verified Meta Business account, a real setup project on its own, while share links work today for free; document and complaint-photo uploads are metadata-only until real file storage is connected.

## The Supabase swap: how it fits together

Unchanged in spirit from session 1: `src/lib/store.tsx`'s `useData()` hook is still the only thing pages touch, and its public shape (the field and action names) hasn't changed, only its internals need to change. What's different now that the SaaS layer exists:

1. **Auth first, using Supabase's built-in email magic link** (Authentication → Providers → Email, "Enable Email OTP" style passwordless sign-in is on by default, no extra provider setup needed, unlike phone). Replace `Login.tsx`'s email step so it actually calls `supabase.auth.signInWithOtp({ email })` instead of just flipping local `sent` state.
2. **The membership-claim step.** Because a resident's `memberships` row is created by the committee (via the Members page, using the email captured there) before that resident ever logs in, the first successful magic-link login needs to match `auth.users.email` to a pending `memberships` row and attach the real `user_id`. Cleanest place for this: a Supabase Edge Function triggered on new `auth.users` inserts, or a client-side "claim" query run once right after first login. Either way, this is new work, session 1's plan didn't need it because it assumed phone OTP with a different membership model.
3. **Read path, table by table**, same order as before: `societies` → `flats` → `notices` → `documents` → `vendors` → `bills`/`payments` → `complaints` → `polls`/`poll_votes` → `events` and its 3 sub-tables → `vehicles`/`contacts`/`adjustments`. The scoping-by-active-society logic already lives in one place in `store.tsx`, so each table's swap is: replace that table's slice of local state with a Supabase query filtered by RLS (no explicit `.eq('society_id', ...)` even needed once RLS is on), keep the shape matching `types.ts` exactly.
4. **Write path, same as session 1's plan**, plus: `addSociety` and `updateSocietyById` (new this session) need real inserts/updates against `societies`, and `updateSocietyById` specifically needs to work regardless of which society is "active" in the caller's session, since the owner console edits societies it isn't necessarily "inside."
5. **Theme and module data already round-trips cleanly**: `themeKey` is just a string column, `modules` is already shaped as a JSON object matching the `modules jsonb` column in `supabase/schema.sql`, no transformation needed either direction.
6. **Logo upload**: swap the onboarding wizard's and society-detail page's `FileReader`-to-data-URL logic for a real upload to the `documents`-style Storage bucket pattern already described in `supabase/README.md`, storing the resulting URL in `societies.logo_url` instead of a data-URL.
7. **Receipt numbering**: same atomicity note as session 1, needs to happen inside the same transaction as the payment insert (`UPDATE societies SET receipt_seq = receipt_seq + 1 WHERE id = ... RETURNING receipt_seq`), now scoped per-society rather than global, which the schema already reflects.

Test each table's swap against `docs/TESTING_PLAN.md`'s four critical flows before moving to the next table.
