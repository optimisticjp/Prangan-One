# Project memory and next steps

This file exists so a future Claude Code session (or a human) can pick this project up without re-deriving everything from the source. Read this before making architectural changes. Updated after each major session; the most recent session is summarized first.

## Project identity

**What this is:** Prangan One, "The Society OS," a Gujarati-first, multi-tenant society management platform. It started as a single-society demo for Rajhans Tower (a real 24-flat society in Surat), which remains the first real society on the platform, created from the owner console like any other, not hardcoded as the only one. Domain: pranganone.com. Support: care@pranganone.com.

**The core stance that shapes every screen:** the committee is the customer and the hero, residents are upside. A society must get full value from the committee side alone, with zero residents logged in. Resident self-service is a second adoption wave, never a dependency for the society to run.

**Current phase:** Almost everything in `docs/PRANGAN_ONE_ROADMAP.md` that can be built without a live Supabase project now exists as real, tested code: the full role model, the subscription lifecycle with a working write-guard, the complete owner backend, real (if locally-scoped) magic-link auth code, CSV import, receipt cancellation, code splitting, an error boundary, and the public marketing site. The schema is fully written for the target state. What's left is fundamentally about a live database existing: connecting it, running the roadmap's own hard checkpoint (a real second society, verified isolation), and then the two things that were always going to need it (file storage, and the membership-claim flow that only makes sense once real auth users exist).

**Who it's for:**
- **Residents:** check their bill, pay, complain, read notices, vote, RSVP to events.
- **Committee (admin):** the treasurer/secretary/president running the society day to day. Almost never a software person, see "Designing for the committee's actual skill level" below.
- **Accountant:** a narrower role focused on the money side and audit exports.
- **Owner:** creates and manages every society on the platform from `/owner/*`. This role can be more technical than the others, since it's the platform owner's own tool, not a committee-facing one.

*(Note on reading the session log below: earlier sessions correctly used "Essancia" when describing who was operating the platform at the time that was written. That's accurate history, left as-is rather than sanitized, the standing description above is what's current now.)*

## Session 8 summary (this session): the actual identity/onboarding architecture, not just login

The person and an external review (ChatGPT, attached in full) both converged on the same real problem: this project had a login mechanism but no actual identity model behind it. This session built that model, end to end, following the "who are you, which society, what role, straight to the right place" architecture both reviews independently landed on, with one deliberate deviation stated clearly below.

**The core resolution, finally wired:**
- `Session` gained `explicitSociety` (already existed) working alongside a real resolution path now: `AuthCallback.tsx` at `/auth/callback` (the route the magic link already pointed at, which genuinely didn't exist as a page until now) waits for the real authenticated user, calls `claimMemberships`, and either lands them straight in the right dashboard (one membership), shows a society picker (more than one), or sends them to the new `/no-access` page (none).
- `claimMemberships` in `auth.ts` now also returns the society's real name (needed for the picker, a raw id isn't something to show anyone), and only claims `status = 'active'` rows, a pending self-enrollment isn't claimable until approved.
- **Deliberately did not put Google login first**, despite the attached review recommending it. Reasoning: this is the one part of the whole project that can't be verified from this sandbox (no network access to supabase.co), adding a second unverified external integration (a separate Google Cloud OAuth setup) on top of the first felt like compounding risk rather than reducing friction. Get email + resolution proven first, add Google next to it once that's confirmed working.

**Resident self-enrollment (`/join`), Google Classroom's class-code pattern, adapted:**
- `Society.joinCode`, a short generated code, shown and copyable from the owner console's society detail page.
- A resident enters the code, their flat number, name, phone, and email. Matched against what's on file for that flat - if the phone matches, they're in immediately; if not, it's a pending request a committee member approves with one tap (new card in `admin/Members.tsx`).
- **This matching decision is enforced server-side**, not just in the local demo's JS. `supabase/schema.sql` got a real `before insert` trigger (`enforce_membership_insert`) that ignores whatever status a self-enrollment request claims and decides active-vs-pending itself by checking the flat's phone on file. Verified this by actually installing Postgres 16 in this sandbox and running the real trigger against real test data, not just reasoning about it, see the specific scenarios below.

**Three real bugs the Postgres testing caught, that would not have been caught any other way:**
1. `Society.slug` (the `/s/:slug` feature) had never actually been added to `schema.sql` at all - the real database was missing a column the app has depended on since that feature was built. `join_code` needed the same column, added both.
2. `memberships` had no `update` policy whatsoever. RLS defaults to deny with no matching policy, meaning `claimMemberships`' own claim-by-email update, and any future approval action, would have silently failed against a real project. Added `memberships_claim` (claim your own pending invite, only ever to your own uid) and `memberships_manage` (committee/owner managing memberships in their society).
3. The self-enrollment trigger's own bootstrap path was initially broken: it checked `has_role()`, which requires an *existing* privileged membership to already exist, exactly the thing that doesn't exist yet when manually bootstrapping the very first owner/admin row through the Supabase SQL editor. Fixed by checking `session_user` (not `current_user`, which reports the function's owner inside a `security definer` function, not the caller) against `postgres`/`service_role`/`supabase_admin`, so a direct database connection bypasses the self-enrollment restrictions while the public API (`anon`/`authenticated` roles) still gets them enforced. Verified this exact distinction with real `SET SESSION AUTHORIZATION` role switches, not just assumed.
4. `societies_select` was member-only, which quietly broke both `/s/:slug` and `/join` against real RLS - a visitor isn't a member of anything yet, so they could never look up a society by slug or code in the first place. Made societies fully publicly readable (nothing in the table is sensitive; all of it already shows on the public app or public share pages anyway).
5. `flats`, deliberately, did NOT get the same treatment - a flat row has owner/tenant names, phone, email, broadening that would leak every resident's contact info to anyone who knows a join code and guesses a flat number. Added a narrow `find_flat_for_join()` function instead: security definer, returns only an opaque id, never the row.

**The "no membership found" page, done as the marketing moment it should be, not a dead end:** `/no-access`, using the more precise wording from the attached review ("we couldn't find a society linked to this email yet," which doesn't imply their society isn't on the platform, just that this particular email hasn't been added) - offers the join-code path and the "bring my society here" path (straight into the existing Contact/Formspree pipe), and logs the attempt (email only, into a new owner-only `unmatched_login_attempts` table) so the owner console shows it as a real interest signal, grouped by email with a count, kept separate from the sales lead inbox since nobody agreed to being contacted about it.

**Owner support-mode refinements**, per the attached review's "require a reason, show a persistent banner" recommendation: `enterSociety` now takes an optional `reason`, required via a modal before entering write-capable support (read-only stays a single click, no reason needed), logged on the `ImpersonationLog` and shown in the owner's Activity page. A persistent, hard-to-miss saffron banner now shows across every admin/accountant screen while the owner is actively in support mode, with a one-click exit.

**What's still explicitly not done, on purpose:** Google login (see above). The full owner-dashboard six-section reorganization the review suggested - scoped out of this pass deliberately, it's an organizational/cosmetic change, not a functional gap, and this session was already large. The actual Supabase-backed version of `selfEnrollResident` (the local demo layer has the real, tested matching logic; the database has the real, tested trigger enforcing it; the JS function that calls the real Supabase insert for this specific flow doesn't exist yet, same bucket as the rest of the "swap the write actions" phase).

- **Module system, both layers.** `Society.modules` is now `{ ownerEnabled, adminVisible }`. - **The owner backend is real**, not a preview. Moved from `/saas` to `/owner/*` (old links redirect, don't 404): a dashboard with the exact stats the roadmap asks for (societies by status, total flats, total memberships, expected monthly revenue, pending platform payments, recent activity), a societies list with inline subscription controls, the onboarding wizard, a per-society detail/edit page with subscription lifecycle buttons and membership management, a manual platform billing tracker (₹10/flat/month, editable per society), a leads inbox, and an activity log showing both the audit trail and every owner impersonation session (read-only vs write-capable, both logged, per the roadmap's non-negotiable).

- **CSV member import, for real.** `src/lib/csv.ts` has a hand-written CSV parser (quoted fields, embedded commas/newlines, escaped quotes) and a row-level validator, both covered by 12 real unit tests. ## Decisions log

**Session 2 additions:**

- **Cloud hosting: Cloudflare Pages, not Vercel.** Verified this directly rather than assume: Vercel's free Hobby plan is restricted by its own terms to non-commercial, personal use, and this becomes a paid client engagement the moment it's set up for Rajhans Tower. Cloudflare Pages' free tier explicitly allows commercial use, with unlimited bandwidth and no credit card required. Same git-deploy workflow either way.
- **Auth: email magic link, not phone OTP.** The person chose this explicitly after being shown the cost tradeoff. Real phone OTP in India needs TRAI DLT sender registration (roughly ₹5,900 + GST one-time, 3 to 7 working days to clear) plus a per-message cost, none of which is actually free. Supabase's built-in email magic link needs no separate provider and costs nothing on the free tier. The tradeoff: this needs residents to have and check an email address, which won't be true for every member, especially older ones. The Members page now collects an optional email per flat (owner and tenant separately) for exactly this reason, and CLAUDE_CODE_NEXT_STEPS keeps a fallback path where the committee can act on a resident's behalf if that resident never gets a working login.
- **Supabase free tier's 7-day auto-pause is a real operational risk**, not a footnote. A free project with no traffic for seven straight days pauses itself until someone manually resumes it from the dashboard. For a live committee's app, a quiet week (a holiday, low adoption at first) would mean the app just looks broken. Fix: a scheduled free keep-alive ping (GitHub Actions cron or an UptimeRobot check) every few days. This needs to be set up as part of the go-live checklist, see supabase/README.md.
- **Not everything is configurable.** Explicitly rejected a fully generic "any field, any color, any layout" system. Societies control branding (curated theme + logo), setup (maintenance amount, due day, receipt numbering), and which modules are on, nothing else. Product structure, page layout, and the core Gujarati copy stay fixed platform-wide. This was the person's own instinct going in, and it's held throughout the build.
- **Designed for the committee's actual skill level, not a typical SaaS admin's.** A Rajhans-style treasurer or secretary is comfortable with WhatsApp and a UPI app, not with SaaS jargon. Concretely: the onboarding wizard defaults everything to sensible values rather than starting blank; internal vocabulary (module, tenant, theme key) never appears in any user-facing copy, only in code comments and this file; the branding step is tap-to-choose preset cards, never a hex input; every module toggle has a one-line plain-language reason, not just a name. The SaaS owner console is the one deliberate exception, it can stay more technical since it's Essancia's own tool.
- **Self-serve-ready architecture, Essancia-operated for now.** The person wants the option of societies signing themselves up eventually, so the onboarding wizard and data model don't assume Essancia is always the one filling in the form. Nothing today exposes that signup flow publicly though: `/saas/*` still sits behind the `owner` role gate. Making it genuinely public is future work (see Roadmap), and would need real validation, spam/abuse protection, and probably a lightweight approval step before a self-signed-up society goes fully active, none of which exists yet.
- **Committee contact management stayed out of scope this session.** There's no admin UI yet for adding/editing the committee contact list (`db.contacts` is still seed-data-only, no `addContact`/`updateContact` action exists in the store). The onboarding wizard only collects a single support phone number on the Society record itself. Flagged here rather than silently skipped, this is a small, contained follow-up whenever it's needed.

**Session 1 decisions** (unchanged, still accurate): localStorage over a mock backend for the first pass so the committee could react to the UI before any backend investment; no gate/visitor security module, explicitly out of scope; spoken Surat Gujarati throughout, not textbook Gujarati; green/amber/red as the only status language app-wide; WhatsApp via `wa.me` share links rather than the Cloud API, since the Cloud API needs a verified Meta Business account, a real setup project on its own, while share links work today for free; document and complaint-photo uploads are metadata-only until real file storage is connected.

## The Supabase swap: how it fits together

Unchanged in spirit from session 1: `src/lib/store.tsx`'s `useData()` hook is still the only thing pages touch, and its public shape (the field and action names) hasn't changed, only its internals need to change. What's different now that the SaaS layer exists:

1. **Auth first, using Supabase's built-in email magic link** (Authentication → Providers → Email, "Enable Email OTP" style passwordless sign-in is on by default, no extra provider setup needed, unlike phone). Replace `Login.tsx`'s email step so it actually calls `supabase.auth.signInWithOtp({ email })` instead of just flipping local `sent` state.
2. **The membership-claim step.** Because a resident's `memberships` row is created by the committee (via the Members page, using the email captured there) before that resident ever logs in, the first successful magic-link login needs to match `auth.users.email` to a pending `memberships` row and attach the real `user_id`. Cleanest place for this: a Supabase Edge Function triggered on new `auth.users` inserts, or a client-side "claim" query run once right after first login. Either way, this is new work, session 1's plan didn't need it because it assumed phone OTP with a different membership model.
3. **Read path, table by table**, same order as before: `societies` → `flats` → `notices` → `documents` → `vendors` → `bills`/`payments` → `complaints` → `polls`/`poll_votes` → `events` and its 3 sub-tables → `vehicles`/`contacts`/`adjustments`. The scoping-by-active-society logic already lives in one place in `store.tsx`, so each table's swap is: replace that table's slice of local state with a Supabase query filtered by RLS (no explicit `.eq('society_id', ...)` even needed once RLS is on), keep the shape matching `types.ts` exactly.
4. **Write path, same as session 1's plan**, plus: `addSociety` and `updateSocietyById` (new this session) need real inserts/updates against `societies`, and `updateSocietyById` specifically needs to work regardless of which society is "active" in the caller's session, since the owner console edits societies it isn't necessarily "inside."
5. **Theme and module data already round-trips cleanly**: `themeKey` is just a string column, `modules` is already shaped as a JSON object matching the `modules jsonb` column in `supabase/schema.sql`, no transformation needed either direction.
6. **Logo upload**: swap the onboarding wizard's and society-detail page's `FileReader`-to-data-URL logic for a real upload to the `documents`-style Storage bucket pattern already described in `supabase/README.md`, storing the resulting URL in `societies.logo_url` instead of a data-URL.
7. **Receipt numbering**: same atomicity note as session 1, needs to happen inside the same transaction as the payment insert (`UPDATE societies SET receipt_seq = receipt_seq + 1 WHERE id = ... RETURNING receipt_seq`), now scoped per-society rather than global, which the schema already reflects.

Test each table's swap against `docs/TESTING_PLAN.md`'s four critical flows before moving to the next table.
