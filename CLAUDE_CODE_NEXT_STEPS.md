# Project memory and next steps

This file exists so a future Claude Code session (or a human) can pick this project up without re-deriving everything from the source. Read this before making architectural changes. Updated after each major session; the most recent session is summarized first.

## Project identity

**What this is:** Prangan One, "The Society OS," a Gujarati-first, multi-tenant society management platform. It started as a single-society demo for Rajhans Tower (a real 24-flat society in Surat), which remains the first real society on the platform, created from the owner console like any other, not hardcoded as the only one. Domain: pranganone.com. Support: care@pranganone.com.

**The core stance that shapes every screen:** the committee is the customer and the hero, residents are upside. A society must get full value from the committee side alone, with zero residents logged in. Resident self-service is a second adoption wave, never a dependency for the society to run.

**Current phase:** Almost everything in `docs/PRANGAN_ONE_ROADMAP.md` that can be built without a live Supabase project now exists as real, tested code: the full role model, the subscription lifecycle with a working write-guard, the complete owner backend, real (if locally-scoped) magic-link auth code, CSV import, receipt cancellation, code splitting, an error boundary, and the public marketing site. The schema is fully written for the target state. What's left is fundamentally about a live database existing: connecting it, running the roadmap's own hard checkpoint (a real second society, verified isolation), and then the two things that were always going to need it (file storage, and the membership-claim flow that only makes sense once real auth users exist).

**Who it's for:**
- **Residents:** check their bill, pay, complain, read notices, vote, RSVP to events.
- **Committee (admin):** the treasurer/secretary/president running the society day to day. Almost never a software person, see "Designing for the committee's actual skill level" below.
- **Accountant:** a narrower role focused on the money side and audit exports.
- **Owner:** creates and manages every society on the platform from `/owner/*`. This role can be more technical than the others, since it's the platform owner's own tool, not a committee-facing one.

## Third plan, build 2 of 3: receipt numbers can no longer collide, and payment recording is now genuinely atomic

The two real findings this build exists for: a receipt number was computed from a number the browser last happened to see, then only ever incremented locally, never saved back to the real database - meaning a refresh between two receipts, or two people recording payments around the same instant, could genuinely produce the same number twice, and nothing in the database would even notice. And recording a payment was two or three separate requests, not one all-or-nothing step, so a bad connection partway through could leave a payment existing while its bill hadn't caught up yet.

**The actual fix: three real database functions, each doing everything in one true transaction, not three or four separate client requests.** One handles recording a fresh payment (including the credit-adjustment logic for an overpayment, computed from the bill's real, currently-locked figures, not whatever the client happened to have cached), one handles confirming a resident's "I have paid" into a real, receipted payment, and one handles cancelling a receipt. All three either fully succeed together or fully roll back together - there's no longer a window where part of this could go through and part of it couldn't.

**The actual fix for the collision itself: the next receipt number is now allocated under a real database lock**, not read-then-hoped-for. Two calls for the same society arriving at the exact same instant can no longer both see the same starting number - the second one genuinely waits for the first to finish and commit before it can even read the counter. This is the single most important thing in this build, so it wasn't just written and trusted: it was tested with two genuinely simultaneous background processes, launched at the same instant, and confirmed to receive two different, correctly sequential receipt numbers, not a theoretical argument about how the locking should behave.

**Who can actually do what was never touched, duplicated, or re-implemented anywhere in this.** These three functions run with the caller's own ordinary permissions, not elevated ones, so the exact same rules that already decided who can record or confirm a payment keep deciding it, completely unchanged. Only the one narrow step of incrementing a society's receipt counter - which an accountant needs to be able to trigger but was never allowed to touch a society's row directly - reaches into a small, separate, purpose-built helper for that one specific piece of bookkeeping. Confirmed directly: a plain resident is still correctly refused from directly recording a successful payment or confirming a pending one, exactly as before, and the real self-enrollment flow that touches an unrelated protected table still works exactly as it did.

**All three genuinely safe to retry, the same discipline every other real write in this app already follows.** Each function checks whether this exact call already happened - a payment id that already exists, a payment that's no longer pending, a receipt that's already cancelled - and returns the existing result instead of doing any of it a second time. Confirmed directly: calling the same payment twice never allocates a second receipt number and never double-applies a bill update.

**The client side reconciles instant feedback with what's actually true, rather than choosing between them.** A receipt number still appears on screen the moment someone clicks record, for the same instant feel the app already had - but it's now explicitly optimistic, and gets corrected to whatever the database actually, atomically allocated the moment that real response comes back. What ends up stored, printed, and shown to anyone who refreshes is always the real number, never the client's guess.

**Added three new permanent, automated scenarios to the isolation test suite itself**, not just relying on the manual verification done while building this - sequential receipt allocation producing genuinely different numbers, retry-safety, and the resident-block confirmation, so these specific properties can't quietly regress later without something catching it.

**Tests:** 239 passing, up from 236. Rewrote the existing `recordPaymentReal` tests to match its new shape (a single RPC call instead of two separate table operations), and added dedicated tests proving the reconciliation logic specifically - that the optimistic number shown immediately for instant feedback gets genuinely replaced by the real one, not left sitting alongside it.

Needs the schema reapplied - this build is centered on real database functions.

## Fifth plan, build 1: the first real CI run found a real bug, exactly the kind this repository could never have caught on its own

Flagged plainly when CI was first built (fourth plan, build 2): this repository has no way to actually execute a GitHub Actions run from inside it, only check that the logic and YAML are sound. The first real push after that build was the actual first proof, and it found a real, genuine bug, not in the app, not in any database policy, in the CI workflow itself.

**What actually happened:** the "type check, tests, build" job passed cleanly. The "database isolation suite" job failed, and the actual error was `connection to server on socket "/var/run/postgresql/.s.PGSQL.5432" failed: No such file or directory` - psql trying to connect over a local Unix socket, one that doesn't exist on that runner at all.

**Traced precisely, not guessed at:** the workflow's own "wait for Postgres" step already explicitly used `-h localhost` and passed cleanly, proving the actual database was up and reachable over TCP the entire time. What failed was the isolation script's own four `psql` calls, which have never specified a host anywhere, correctly, since that's what makes them work against a locally-running Postgres process - true in this project's own sandbox, true on a real developer's own machine, but not true in GitHub Actions' own `services:` feature, which runs Postgres in a separate, sibling container, reachable only over TCP, never through a shared filesystem socket.

**The actual fix, and why it's shaped the way it is:** `PGHOST: localhost` set for that one CI step specifically, in the workflow file alone. The shared script itself (`run-isolation-tests.sh`) was deliberately left completely untouched - it's correct everywhere else it runs, including this project's own sandbox verification process, and hard-coding a host into the script itself would have risked breaking that in order to fix something that's actually specific to how GitHub Actions' service containers work.

**Verified as thoroughly as this repository is able to, given it still cannot execute a real GitHub Actions run directly:** confirmed the underlying mechanism directly, not just the YAML syntax - added a genuine trust rule for TCP connections to this sandbox's own Postgres configuration, matching exactly what `POSTGRES_HOST_AUTH_METHOD: trust` does in the real Postgres Docker image CI actually uses, then ran the complete, entirely unchanged isolation script over a real TCP connection with `PGHOST` set, the exact same way the fixed workflow now does. All 40-plus scenarios passed. This is the most faithful proof available without GitHub's own infrastructure - genuine confidence, not certainty, and this next push is what actually confirms it.

No app code changed in this build, and no further database changes - purely the one CI workflow file.

## Fourth plan, build 3 of 7: the demo's real foundation - a genuinely separate provider, not a flag inside the real one

The finding this build responds to, and the reason it's the largest, riskiest piece of this whole plan: the public demo and the real, live product shared one file, one storage key, one set of write functions, with a runtime check (`isRealSession`) deciding which path each of them took. That check had already been proven correct, exhaustively, action by action, in earlier work on this project - but "proven correct by reading the code carefully" and "structurally impossible to get wrong" are two different kinds of safety, and this build is about building the second one, not re-proving the first.

**The real change: the demo now lives in its own file, and that file genuinely cannot reach Supabase, not because a flag tells it not to, but because it never imports the modules that could.** `demoStore.tsx` never references `realData.ts` or `lib/supabase.ts` anywhere in its own source, and neither does anything it depends on. Confirmed two different ways, not one: a static test that reads the actual file text and fails if that reference ever appears again, and a runtime test that mocks the real modules entirely and runs a full demo session - role picked, payment recorded, complaint filed, notice posted, poll voted on - confirming none of it was ever called, not even once.

**Extracted the shared parts first, the pieces both providers now genuinely reuse identically rather than each having their own copy.** The pure, derived-data math (what's overdue, what a flat still owes, this month's income) moved to its own file, taking no side, closing over nothing - both providers hand it their own state and get the same, correct answer back. The shared contract every page depends on (the `Store` interface, and the `useData()` hook every single page already calls) moved to a third, neutral file that neither provider owns, which is what makes the separation real: the demo provider imports from that neutral file, the real provider imports from that same neutral file, and neither one imports from the other at all.

**Storage is genuinely separate too, not just the code.** The demo's own data lives under its own key prefix (`prangan_demo_v1_*`), in `sessionStorage`, not `localStorage` - the spec's own stated preference, and a real, additional safety property: a demo session clears itself the moment the tab closes, rather than sitting around on someone's laptop indefinitely the way the real session's own storage does. A real session's cached data can never bleed into a demo session on the same browser, or the other way round, even by accident.

**Every id in the demo is now genuinely, unmistakably synthetic** - `demo-society-001`, `demo-flat-a101`, and so on, both in the seed data and for anything created fresh during a session, never the shape a real or locally-cached record would ever have.

**The owner console is no longer offered in the demo at all**, per an explicit, named requirement in the spec this was built against. Removing it surfaced a real, separate bug while doing it: the fetch effect's own idea of which society to load had a silent fallback that could substitute a completely different society if the intended one wasn't already sitting in the browser's local cache - fragile, incidental protection rather than an explicit guarantee, and exactly the kind of thing this whole project has been about not leaving to chance. Fixed at the root: the fetch now always uses the session's own explicit, intended society id, never a value that could silently fall back to the wrong one. This fix benefits every real session too, not just the code path that surfaced it.

**How switching between the two providers actually works, and why it deliberately isn't a live swap under one already-mounted tree:** picking a role on `/demo` writes the choice to the demo's own storage, then forces a genuine, full page reload to the destination - not a client-side navigation. A small, single check at the very start of the app's own startup (`main.tsx`) reads that storage, synchronously, before the very first render, and mounts the real provider or the demo provider accordingly. Two structurally separate React trees that never coexist under one parent, rather than one already-running tree trying to swap out its own foundation live - simpler to reason about, and there was no real cost to choosing the safer option here, since a role pick was never a moment that needed to feel instantaneous the way an in-app action would.

**What this build deliberately does not include, stated plainly rather than left to be discovered later:** the seed data here is minimal, enough to prove the architecture end to end, not the fuller "Sahaj Residency" fictional society with realistic disorder called for later in this same plan - that content replaces this file's contents directly when it's built, without needing to touch the provider itself. Most of the 44 real, mutating actions this project's store exposes have straightforward, working local implementations in the demo now, ported from the same, already-correct logic the real provider uses, not rewritten from scratch - but the deeper, connected committee-to-resident payment journey called for in this plan is proven at the level of "the actions work and stay local," not yet built out as its own guided, connected experience. That's genuinely the next build's job, not something this one reached for prematurely.

## New plan, build 5 of 6: internal database helper functions are no longer reachable as API calls, and the first fix attempted for this didn't actually work

The finding: twelve internal helper functions (things like `has_role`, checking whether the current person holds a given role in a given society) lived in the same database schema, `public`, as everything else. **The actual fix: these twelve functions moved to their own schema, `private`, which Supabase's API layer never exposes at all.** Real Row Level Security policies (the actual thing enforcing every access rule in this app) can still reach them completely normally, since Postgres doesn't restrict a policy from referencing a function in a different schema, only the connecting role's own privileges matter for that. The three functions genuinely meant to be called directly by the app - the public society lookup, poll results, and the join-request flow - stayed exactly where they were.

**Verified, not assumed, that the legitimate flow still works from the other side of this.** The actual self-enrollment flow, `submit_join_request`, calls three of these now-relocated functions internally. **Worth being honest about: the first version of this fix looked right and wasn't.** Moving a function to a new schema doesn't, on its own, remove Postgres's own default behavior of granting every new function's execute privilege to everyone. The first attempt used `ALTER DEFAULT PRIVILEGES` to try to prevent that default from applying going forward, and it read as correct, tested as inconclusive, and turned out not to actually be taking effect - caught only by testing an actual direct call as a real, restricted role and watching it wrongly succeed, not by trusting the statement's own syntax. ## New plan, build 3 of 6: a failed save now genuinely survives a reload, not just a toast that disappears

The design discussion for this build was real, even though it happened quickly: a failed write already showed a toast with a retry button, but that toast lived only in memory. Dismiss it, or just reload the page, and the warning vanished while the actual unsaved record kept sitting there in local storage looking exactly like it had saved fine. The fix had to make "this hasn't been confirmed saved" something that survives a reload, not something shown once and then trusted to memory.

**The core mechanism: `db.pendingSync`, persisted the same way every other real field already is.** Every one of the 33 real writes across the app now records a pending entry there on failure, and clears it on success - not a second, parallel tracking system, the same `attemptRealWrite` function every write already goes through. `failedWrites` (what the existing banner already reads) is now a live view of this persisted list, not separate, memory-only state - the banner component itself needed no changes at all, it was already built correctly, it just needed a durable thing to look at.

**The harder problem: a retry after reload can't use the original closure, because a function can't survive being saved to and read back from local storage.** Solved by rebuilding the retry from the record's own current data instead - every insert this touches was already made safe to attempt again in an earlier build (each one checks whether its row already exists first, or is naturally idempotent), so looking the record up fresh in `db` and calling the same real-write function again is both correct and safe, even if part of the original attempt had actually gone through.

**A genuinely tricky case, handled honestly rather than papered over.** A handful of writes (poll votes, event contributions/volunteers/expenses, batch bill generation) use a composite string id instead of a real record id, like `vote-{pollId}-{flatId}`. Parsing that back apart naively would be unreliable, since real ids are UUIDs that already contain hyphens. Added a small `context` field to carry exactly what's needed for these specific cases instead of guessing from a string. One case - a single event expense line item - genuinely can't be safely auto-resynced at all, since it has neither a client id nor a natural way to tell "already saved" apart from "a second, different expense with the same label." Rather than risk a real duplicate, that one specific kind is tracked and shown, just not silently retried - written down plainly in the code, not left looking as solid as everything around it.

**Tested against the actual failure mode, not a simulation of it.** The critical test here doesn't call a function twice - it renders a real component tree, fails a write, genuinely unmounts the entire tree (not just resetting a variable), then mounts a completely fresh one that has only localStorage to read from, exactly like a real page load. ## Build 3 of 9 (this session): real writes can now fail visibly, with a genuine retry, not silently

The audit's single largest finding: 33 places where a real write's failure was invisible, the screen would show the change as if it had saved and nothing further would happen. This build establishes the actual mechanism and proves it on the three highest-stakes write paths - payments, bill generation, and complaints. Build 4 rolls the same, now-proven pattern out to the remaining paths.

**Chose a real retry button everywhere, not a split between money and everything else.** Discussed this directly before writing code - the option to give complaints and notices a simpler rollback-plus-error treatment while reserving a real retry button for payments was on the table, but the call was to keep one consistent behavior across every real write rather than maintain two different patterns.

**The core mechanism: `attemptRealWrite`, one function every real write now goes through.** Tracks failures by the item's own id, so a specific failed complaint or payment can be retried on its own, not a generic "something failed" with no way to know what. A retry calls the exact same operation again, not a different code path - genuinely the same write, not a refetch or a reset.

**A real correctness problem found and fixed while building this, not after.** A naive retry of a multi-step write (insert, then a second step like a photo upload) would re-attempt the insert too - and for a resident filing their own complaint or recording their own payment, that insert can't simply be upgraded to an upsert, since upsert also needs UPDATE permission, which only society_admin/accountant actually have on those tables. Fixed by checking whether the row already exists before inserting, rather than assuming upsert is always safe - verified this exact scenario directly against real Postgres: a real resident's complaint, a simulated retry, confirmed no duplicate-key error and no duplicate timeline entry. Bill generation needed the same care for a different reason: it inserts a batch, and a naive upsert there risked silently overwriting a bill's paid_amount if a resident had paid it in the time between an original attempt and a stale retry - fixed by filtering out whichever bills in a batch already exist, checked in one query, before inserting the rest.

**The UI: a persistent banner, not the existing auto-dismissing toast.** The blocked-write toast auto-dismisses after 4 seconds, which is correct for a message that doesn't need action. This one deliberately doesn't auto-dismiss - the entire point is giving someone time to notice a failure and tap retry, and a message that vanishes before they can act on it would defeat that. Shows up consistently across the resident, committee, and owner consoles.

**Tests:** 203 passing, up from 198. Covers the mechanism itself directly - a failure surfaces, a successful retry clears it, a first-try success never appears at all, dismissing doesn't trigger a hidden retry, and the local demo never touches this mechanism at all, confirmed with a real, hoisted mock of the actual write function rather than just checking the surrounding logic.

## Decisions log

**Session 2 additions:**

- **Cloud hosting: Cloudflare Pages, not Vercel.** Verified this directly rather than assume: Vercel's free Hobby plan is restricted by its own terms to non-commercial, personal use, and this becomes a paid client engagement the moment it's set up for Rajhans Tower. Cloudflare Pages' free tier explicitly allows commercial use, with unlimited bandwidth and no credit card required. Same git-deploy workflow either way.
- **Auth: email magic link, not phone OTP.** The person chose this explicitly after being shown the cost tradeoff. Real phone OTP in India needs TRAI DLT sender registration (roughly ₹5,900 + GST one-time, 3 to 7 working days to clear) plus a per-message cost, none of which is actually free. Supabase's built-in email magic link needs no separate provider and costs nothing on the free tier. The tradeoff: this needs residents to have and check an email address, which won't be true for every member, especially older ones. The Members page now collects an optional email per flat (owner and tenant separately) for exactly this reason, and CLAUDE_CODE_NEXT_STEPS keeps a fallback path where the committee can act on a resident's behalf if that resident never gets a working login.
- **Supabase free tier's 7-day auto-pause is a real operational risk**, not a footnote. A free project with no traffic for seven straight days pauses itself until someone manually resumes it from the dashboard. For a live committee's app, a quiet week (a holiday, low adoption at first) would mean the app just looks broken. Fix: a scheduled free keep-alive ping (GitHub Actions cron or an UptimeRobot check) every few days. This needs to be set up as part of the go-live checklist, see supabase/README.md.

Societies control branding (curated theme + logo), setup (maintenance amount, due day, receipt numbering), and which modules are on, nothing else. ## The Supabase swap: how it fits together

Unchanged in spirit from session 1: `src/lib/store.tsx`'s `useData()` hook is still the only thing pages touch, and its public shape (the field and action names) hasn't changed, only its internals need to change. What's different now that the SaaS layer exists:

1. **Auth first, using Supabase's built-in email magic link** (Authentication → Providers → Email, "Enable Email OTP" style passwordless sign-in is on by default, no extra provider setup needed, unlike phone). Replace `Login.tsx`'s email step so it actually calls `supabase.auth.signInWithOtp({ email })` instead of just flipping local `sent` state.
2. **The membership-claim step.** Because a resident's `memberships` row is created by the committee (via the Members page, using the email captured there) before that resident ever logs in, the first successful magic-link login needs to match `auth.users.email` to a pending `memberships` row and attach the real `user_id`. Cleanest place for this: a Supabase Edge Function triggered on new `auth.users` inserts, or a client-side "claim" query run once right after first login. Either way, this is new work, session 1's plan didn't need it because it assumed phone OTP with a different membership model.
3. **Read path, table by table**, same order as before: `societies` → `flats` → `notices` → `documents` → `vendors` → `bills`/`payments` → `complaints` → `polls`/`poll_votes` → `events` and its 3 sub-tables → `vehicles`/`contacts`/`adjustments`. The scoping-by-active-society logic already lives in one place in `store.tsx`, so each table's swap is: replace that table's slice of local state with a Supabase query filtered by RLS (no explicit `.eq('society_id', ...)` even needed once RLS is on), keep the shape matching `types.ts` exactly.
4. **Write path, same as session 1's plan**, plus: `addSociety` and `updateSocietyById` (new this session) need real inserts/updates against `societies`, and `updateSocietyById` specifically needs to work regardless of which society is "active" in the caller's session, since the owner console edits societies it isn't necessarily "inside."
5. **Theme and module data already round-trips cleanly**: `themeKey` is just a string column, `modules` is already shaped as a JSON object matching the `modules jsonb` column in `supabase/schema.sql`, no transformation needed either direction.
6. **Logo upload**: swap the onboarding wizard's and society-detail page's `FileReader`-to-data-URL logic for a real upload to the `documents`-style Storage bucket pattern already described in `supabase/README.md`, storing the resulting URL in `societies.logo_url` instead of a data-URL.
7. **Receipt numbering**: same atomicity note as session 1, needs to happen inside the same transaction as the payment insert (`UPDATE societies SET receipt_seq = receipt_seq + 1 WHERE id = ... RETURNING receipt_seq`), now scoped per-society rather than global, which the schema already reflects.
